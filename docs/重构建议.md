我认真思考过当前整体架构是否过于依赖固定 workflow，是否限制了多 agent 架构本身的灵活性。经过这一轮重构后，当前实现已经在“保留 agent 分工”和“用 Python 控制面保证安全边界”之间形成了更清晰的折中：agent 负责创作、判断和产物生成；Python runtime 负责文件协议、可见性、schema、intent 生命周期、快照、机械交付和前端同步。

本文档现在不再只是理想设计草案，而是对当前实现的结构化说明，并标出仍可继续演进的方向。

## 当前实现总览

当前核心链路是：

```text
browser/server
-> round_prepare / agent packet
-> input analysis
-> GM / projection / player / character / subGM runtime
-> story
-> critic
-> postprocess
-> delivery / handler / memory / lifecycle cleanup
```

主要 agent 与控制面角色如下：

- `input-analyst agent`：分析双通道输入，产出结构化 `input_analysis.output.json`。
- `GM agent`：主线剧情推进、非重要 NPC 扮演、actor/subGM 调度决策。
- `subGM agent`：被 GM 创建和约束的支线剧情推进者。
- `player agent`：玩家角色的第一人称沉浸式 actor。
- `character agent`：重要角色的第一人称沉浸式 actor。
- `projection`：当前主要是 Python 控制面的可见性投影/过滤机制，不是自由协商型 LLM agent。
- `story agent`：只负责把可见剧情整理成第二人称小说正文和来源支撑的角色对话元数据。
- `critic agent`：唯一正文质量门，只审核正文质量、文风、字数、连贯性、用户指令满足度和修复路由。
- `postprocess agent`：critic 通过后生成正文以外的前端数据，包括摘要、行动建议、当前目标、状态 patch、状态面板、MVU 变量更新命令和 UI 扩展数据。
- `assets-ui agent / assets_task`：异步 UI/图片增强方向；当前控制面支持非阻塞 `assets_task` 记录与 deferred 状态，也支持结构化 UI schema 更新时同步存档级 `postprocess_contract.json`。
- `Claude Code 主 agent`：编排脚本运行、代码修改、验证、人工授权的源码修复等工程任务。

## 1. 控制面与 agent 灵活性的边界

当前实现没有把所有交互都写死成单个大 workflow，但也没有允许 agent 任意读写文件或任意互发消息。实际边界是：

- `agent_messages.py` 提供 append-only 消息总线和 per-agent inbox 投影。
- `agent_intents.py` 提供可执行 intent 生命周期：pending / accepted / completed / blocked / rejected。
- `agent_dispatcher.py` 是当前默认 live path 的执行入口，显式消费 pending intents。
- `.agent_runs/<round>/artifacts/` 是控制面权威产物目录；根目录同名 JSON 多数只是交付、记忆或调试边界导出。
- Python runtime 负责 ACL、可见性投影、schema gate、snapshot、repair routing 和 delivery gate。

这意味着系统仍保留多 agent 架构的分工和可扩展性，但生产路径不会让任一 agent 直接绕过控制面写入交付产物。

## 2. agent 间通信的当前形态

原设想中“带收件目标 agent 的文件邮箱”已经部分落地，但形式不是 MCP 工具，而是 repo 内部文件协议：

- `messages.jsonl` 是每轮 append-only 通信日志。
- `inboxes/` 是按 ACL/投影生成的 agent 可读索引。
- `intents/` 是控制面请求队列。
- actor-facing 消息必须先经过 projection，才能进入 player/character inbox。

后续如果要进一步提升 agent 自由度，可以考虑把常用消息发送、投影读取、intent 创建封装为更清晰的工具接口；但生产逻辑仍应保留 Python schema 和权限边界，避免 agent 直接改写权威状态。

## 3. input-analyst agent 的实际职责

当前 input analyst 是入口分析者。它的核心职责不是替下游 agent 产出最终交付物，而是把玩家输入拆成受控、可验证、可路由的结构化意图。

当前已落地职责：

- 接收并分析 `role_channel` 与 `user_instruction_channel`。
- 产出通过校验的 `input_analysis.output.json`。
- 显式声明语义单元、可见性、重要角色、修订/回滚意图等结构化信息。
- `input_analysis_apply.py` 只接受已校验结构，不从玩家原文用关键词、子串或正则推断语义。
- 输入分析应用后会重建 routed agent packets。

需要补充区分两类完全不同的路径：

1. 错误导致的自我修复：critic、delivery 或 runtime 发现产物失败后触发，受 `selfRepairMode` 控制；如果要进入源码修复，还必须同时满足 `allowSourceCodeSelfRepair: true`。
2. 用户主动请求的新能力或存档操作：用户通过指令通道要求系统执行当前未覆盖或未完全自动化的功能时，这不是“错误自我修复”，不应要求 `selfRepairMode` 授权。input analyst 应分析该请求，生成结构化路由决策，再由 dispatcher / Claude Code 主 agent / 相关 agent 在受控边界内执行或阻断。

这一路径的目标能力包括：

- 请求 `assets-ui agent` 更新 UI 数据、调整界面元素或创作图片。
- 主动与 story / GM 讨论是否需要修改、回滚或重演前文剧情。
- 对当前存档文件夹内的角色资料、记忆、关系、变量等进行受控修改。
- 将“需要源码实现的新功能”反馈给 Claude Code 主 agent，进入受限工程修改流程。

边界也需要明确：

- 非源码类的用户主动请求不依赖 `selfRepairMode`。
- 涉及源码修改时，即使请求来自用户指令通道，也必须要求 `allowSourceCodeSelfRepair: true`；如果该开关关闭，应记录为需要授权/无法自动执行，而不是静默修改代码。
- input analyst 不应直接写权威 artifact 或直接修改源码；它应输出结构化 `routing_requests[]` / `analysis_actions[]`，由控制面执行、审计、回滚或阻断。
- 路由决策仍必须来自结构化输入分析，不得在生产代码中用关键词、子串或正则从玩家原文推断语义。

## 4. GM agent 与主线推进

当前 GM agent 仍是主线主持人，拥有全知视角，负责：

- 推进主线剧情。
- 扮演非重要 NPC。
- 决定是否调用 player / character actor。
- 创建、推进、暂停、合并或关闭 subGM 支线。
- 通过 `character_promotions` 请求把非重要角色提升为重要角色。

当前边界：

- GM 不能扮演玩家角色或已注册重要角色。
- GM/subGM 的 `actor_calls[]` 必须携带可见性证明 `visibility_basis`。
- GM 输出进入 actor/story-facing 字段前，会经过隐藏信息清理。
- GM 在 `stop_reason: "complete"` 前必须处理仍活跃的 subGM 支线，不能把 unresolved side thread 留给交付阶段。

## 5. projection 的当前实现

原设想中的 projection agent 更像一个可与 GM/subGM 协商的审核员。当前实现更保守：

- projection 主要由 Python 控制面和 `agent_projection.py` / `agent_actor_runtime.py` 执行。
- 它根据 structured visibility basis、actor 身份、位置、公开/私有范围等确定 actor 可见内容。
- 它会过滤 GM-only、隐藏事实、未证明可见的事件和不安全 metadata。
- 它不依赖 LLM 自由判断，也不会主动与 GM/subGM 协商改写。

后续建议：如果要引入真正的 projection LLM agent，应先保持当前 deterministic projection 作为硬 gate，再把 LLM projection 限制为“提出修订建议”或“补充可读摘要”，不能替代硬权限判断。

## 6. player / character agents

当前 player 和 character agents 都按沉浸式第一人称 actor 运行：

- 只能从投影后的主观上下文感知世界。
- 不直接读取 GM-only 隐藏事实。
- 不直接互相通信；对话转交必须经 GM/subGM 和 projection 边界。
- 输出结构化事件，如 `dialogue`、`custom_action`、`memory_deltas`。

当前 player 与 character 的主要差异：

- player 不参与 subGM 支线。
- player 产生 `high` / `critical` 风险 `custom_action` 时，本轮会停在真实玩家决策点。
- 关键行动证据会进入 postprocess；postprocess 必须把该行动作为固定确认选项之一，字段为 `source: "player_agent_critical_action"` 且 `requires_confirmation: true`。

需要修正原设想中的一句话：不是“player agent 无法作出关键行动”，而是 player agent 可以提出关键行动；系统会把它转化为真实玩家确认点，而不是自动落盘执行。

## 7. subGM agent 与支线

当前 subGM 已作为 GM 的支线模型落地：

- 每条支线有独立 `side_threads/<thread_id>/state.json`、`messages.jsonl`、`interaction.trace.json` 和可选 agent 产物。
- subGM 可读取支线需要的全知信息，但只能在 GM 分配的边界内行动。
- subGM 不能包含玩家角色，不能提升重要角色，不能再派生 subGM，不能直接修改边界。
- subGM 可以向 GM 发送消息，GM 可执行 `accelerate`、`pause`、`resume`、`merge`、`close`。
- 同一角色不能安全参与并行冲突支线；控制面会在调度前检查冲突。

当前落地状态比原设想更严格：支线自由度存在，但必须受 thread state、角色占用和 GM 仲裁约束。

## 8. 重要角色提升与角色资料修改

当前重要角色提升可由两类来源触发：

- input analysis preprocess：写入玩家权威档案。
- GM：写入非玩家权威的 promotion seed，且不能覆盖已有 preprocess/player profile。

subGM 不能创建或提升重要角色，只能通过 `promotion_requests` 或 `boundary_requests` 留给 GM。

角色资料、人设、记忆的修改仍应通过结构化输入分析、GM promotion、actor memory delta、post-round memory summary 等受控路径完成，不应让任意 agent 直接编辑角色文件。

## 9. story agent 的当前职责

story agent 当前只负责正文：

- 读取 `story.input.json`。
- 把 GM/actor/subGM 的可见剧情整理为玩家第二人称小说正文。
- 保留来源支撑的 `character_dialogues` 元数据，用于前端独立对话框。

story agent 当前不负责：

- `<summary>`。
- `<options>`。
- 当前目标。
- 状态面板。
- UI 扩展数据。
- 前端行动建议。
- MVU 变量更新命令。

旧 `<summary>` / `<options>` 解析仍保留为历史 chat log 和 legacy fallback 兼容，但 live story prompt 明确禁止 story 输出这些标签。

## 10. critic agent 的当前职责

critic agent 是唯一正文质量门，但它不再是前端数据提供者。

当前 critic 审核：

- 上下文连贯性。
- 正文质量。
- 文风匹配。
- 字数/长度是否满足。
- 用户指令满足度。
- 玩家输入是否被尊重。
- 角色对话是否有来源支撑。
- 修复路由 `repair_routing`。

当前 critic 不审核、不生成、不修复：

- summary。
- options。
- current_goal。
- state_patch。
- status panels。
- UI extension data。

原第 12 点中“critic 通过后负责提供三个行动建议选项”的设计已经被替换：这些内容现在由 postprocess agent 负责。

## 11. postprocess agent 的当前职责

postprocess 是本轮新增并已落地的关键环节，执行时机是：

```text
critic pass -> run_postprocess -> deliver_round
```

postprocess 输入包括：

- `story.input.json`
- `story.output.json`
- `critic.report.json`
- `interaction.trace.json`
- `ui_manifest.json`
- `postprocess_contract.json`
- generated asset metadata
- pending postprocess repair queue
- current state.js values

postprocess 输出 `postprocess.output.json`，核心字段为：

- `core.summary`
- `core.options`
- `core.current_goal`
- `core.state_patch`
- `ui_extensions`
- `ui_extension_status`
- `mvu.commands`
- `repair_requests`
- `metadata`

交付规则：

- `postprocess.output.json.core` 无效会阻断 delivery。
- UI extension 缺失或失败不阻断正文，但必须写入 repair artifact 和 `.agent_runs/postprocess_repair_queue.jsonl`。
- pending repair 会在下一轮 `input.json.postprocess_repairs` 中暴露，等待下一轮补齐。
- progress bar 和 reply 状态仍归 `progress.json` / `round_state.py`，不属于 postprocess。

## 12. delivery gate 的当前职责

当前 delivery 已经收敛为机械交付，不做正文质量判断。

`round_deliver.py` / `agent_outputs.prepare_delivery()` 会：

- 重建或校验 `story.input.json`。
- 校验 `story.output.json`。
- 校验 `critic.report.json`。
- 在 critic pass 后校验 `postprocess.output.json.core`。
- 把 story 正文镜像到 `skills/styles/response.txt`。
- 调用 `handler.py`。
- ingest memory delta / scheduled memory summaries。
- 执行 post-round memory job 和 lifecycle cleanup。

delivery 不再因为字数、文风、NSFW 或人称发起内容重写；这些属于 critic 的正文质量判断。

## 13. 前端数据的当前来源

当前前端正文和正文外数据来源已经分离：

- 正文：`story.output.json.content` -> `response.txt` -> `handler.py` -> `content.js`
- 剧情摘要：postprocess `core.summary`
- 下一步行动建议：postprocess `core.options`
- 当前目标：postprocess `core.current_goal` 或 `core.state_patch.quest`
- 状态 patch：postprocess `core.state_patch`
- MVU 变量更新：postprocess `mvu.commands` -> `handler.py` -> chat log `variables`
- UI 扩展数据：postprocess `ui_extensions` -> `window.POSTPROCESS_UI`
- 进度条 / 回复状态：`progress.json` / `round_state.py`

handler 会优先读取当前 run 的 `postprocess.output.json` / `artifacts/postprocess.output.json`，再回退到 card root 的 `postprocess.output.json`，避免 stale card-root 数据覆盖当前回合。存在有效 postprocess 输出时，handler 只执行 `postprocess.output.json.mvu.commands` 中的 MVU 命令，不再从 story 正文抽取变量命令；旧正文标签解析只保留为 legacy fallback。

## 14. 浏览器预设选项的当前作用点

当前仍保留并生效的主要运行时设置：

- `style`：作为 story 创作指引，同时进入 critic style alignment 检查。
- `wordCount`：作为 GM/story 的软性目标，并由 critic 的 deterministic quality metrics 做长度检查；玩家真实决策点可豁免扩写。
- `nsfw`：只作为 GM/subGM/story 创作语气提示，不进入 critic 校验。
- `selfRepairMode`：控制 critic revise/block 和 delivery failure 等“错误自我修复”范围；不控制用户通过指令通道主动提出的新能力/存档操作请求。
- `allowSourceCodeSelfRepair`：源码修改独立开关；错误自我修复进入源码修复时仍需 `selfRepairMode == "full"` 且该开关为 true；用户主动请求源码实现新功能时不需要 `selfRepairMode`，但仍必须要求该开关为 true。
- `modelDebugMode`：服务端/前端调试开关，独立保存，不进入 story quality gate。

已删除或不再作为独立全局设置的项：

- 人称。
- 防抢话。
- 背景 NPC。
- 玩家角色名。

当前语义：

- 角色通道默认按玩家角色第一人称输入处理。
- 指令通道默认按第三人称/上帝视角系统指令处理。
- 最终正文默认第二人称。
- player agent 与 character agent 一样沉浸式第一人称行动；“抢话”不再被全局禁止，关键行动会转化为玩家确认点。
- GM 可按剧情需要引入普通 NPC 或请求重要角色提升。
- 玩家角色名应通过角色通道自述或指令通道结构化修改，而不是单独设置项。

## 15. assets-ui agent 的当前状态

assets-ui 方向当前仍是异步增强，而不是正文交付前的硬依赖：

- 图片生成和 UI 热更新不阻塞正文交付。
- dispatcher 可记录非阻塞 `assets_task`，在无外部 worker 时标记为 `deferred`。
- postprocess 会为正文外 UI 数据提供核心支持；如果 UI extension 数据不完整，会进入异步 repair queue。
- `assets_task` payload 若携带结构化 `ui_schema`，dispatcher 会写入存档级 `ui_manifest.json.ui_schema`。
- 同一 payload 若携带 `postprocess_contract`，dispatcher 会同步写入/合并 `postprocess_contract.json`，使下一次 `run_postprocess` 的 Runtime Input 能看到新增 UI 数据需求。
- 若 `ui_schema.postprocess_data_required` 声明了新增数据需求但 contract 缺失或不完整，dispatcher 会写入 `scope: "postprocess_contract"` 的 `.agent_runs/postprocess_repair_queue.jsonl`，本轮不阻塞正文交付，下一轮再尝试补齐。

## 16. 快照、回滚与自我修复

当前每轮准备前会创建 before-round snapshot，用于需要回退本轮派生产物时恢复。

critic repair routing 只覆盖“错误导致的自我修复”路径，当前按失败来源分流：

- `story_composition`：重跑 story/critic。
- `delivery_gate`：机械交付问题，重跑 story/critic 或阻断。
- `gm_loop` / `actor_agent` / `subgm`：仅在 `full` 模式下回退本轮 dispatcher 派生产物，重新调度 GM/actor/subGM。
- `system_code`：必须同时满足 `selfRepairMode: "full"` 与 `allowSourceCodeSelfRepair: true`，才创建受限源码修复请求。

当前不做单个 actor/subGM 的局部热修补，以避免 `interaction.trace.json`、`source_call_id` 与 side thread 状态不一致。

用户在指令通道主动提出的新能力、存档编辑、UI/图片变更、剧情回滚/重演或源码实现请求，不属于上述 repair routing。它应走 input analyst 的用户请求路由，由结构化 `routing_requests[]` / `analysis_actions[]` 表达，并交给控制面执行。该路径可复用 snapshot 与 rollback 能力，但授权语义不同：非源码请求不要求 `selfRepairMode`；源码请求仍要求 `allowSourceCodeSelfRepair`。

## 17. 与原始设想相比的主要调整

已经按设想落地或接近落地：

- 多 agent 文件运行时。
- GM/subGM/player/character 分工。
- projection 可见性边界。
- critic 作为唯一正文质量门。
- story 不再负责 summary/options。
- postprocess 负责前端正文外数据。
- UI extension 失败异步 repair。
- 源码修改独立开关。
- 删除人称、防抢话、背景 NPC、玩家角色名等全局设置。

与原始设想不同、需要按当前实现理解：

- input analyst 当前实现仍偏保守，尚未完整落地“用户请求新能力”的灵活路由；目标形态是让它输出受控 `routing_requests[]`，而不是直接写 artifact 或源码。
- projection 当前不是 LLM 协商 agent，而是 deterministic control-plane projection。
- critic 不提供行动建议；行动建议属于 postprocess。
- story preflight 作为独立质检环节已不再是 live path 的核心质量门；正文质量统一交给 critic，delivery 只做机械 gate。
- player agent 可以提出关键行动；系统会把它变成真实玩家确认点，而不是简单禁止。
- postprocess 不在 `round_prepare.py` 初始 manifest expected outputs 中，而是在 critic pass 后动态调度。

## 18. 后续建议

1. 保持 Python 控制面的硬边界，不要为了 agent 自由度取消 schema、ACL、projection 和 intent 生命周期。
2. 可以把消息发送、intent 创建、投影读取封装成更易用的内部工具，但不要让 agent 直接写权威 artifact。
3. 若引入 LLM projection agent，应把它放在 deterministic projection 之后作为解释/建议层，而不是替代硬 gate。
4. assets-ui 与 postprocess 的 schema 应共同演进：UI schema 变更必须同步 `postprocess_contract.json`；同步失败时必须进入 postprocess repair queue，而不是阻塞正文交付。
5. 为 input analyst 增加用户请求路由 schema，例如 `routing_requests[]` / `analysis_actions[]`，类型可包括 `assets_ui_task`、`story_retcon_consult`、`card_data_edit`、`source_feature_request`，并要求携带目标、payload、授权需求和来源证据。
6. 用户请求的剧情回滚/重演应复用现有 snapshot 与 dispatcher rollback 能力；但不要把它误归类为 critic repair，也不要要求 `selfRepairMode`。
7. 用户请求的源码实现应进入 Claude Code 主 agent 的受限工程流程；不要求 `selfRepairMode`，但必须以 `allowSourceCodeSelfRepair` 为硬门控。
8. 对 live path 中仍保留的 legacy helper 或历史标签解析，应继续限制在兼容/回归测试范围，避免重新成为控制面权威。
