整体架构上，需要围绕AI角色扮演的功能，发多agent架构协作的灵活性，不要过于强调固定workflow的流程，更多地以prompt引导模型，而不是以固定的控制面脚本来实现流程控制。在我的预想中，应该可以保留现有agent架构的同时，大幅度精简系统代码，理清流程中各个agent和python脚本的作用，杜绝重复的环节，提升系统的效率。


1.核心流程是input-analyst agent、GM agent、story agent、critic agent、postprocess agent、delivery gate。
其中GM agent处为核心创作环节，此处实际上还会涉及到subGM agent、player agent、character agent、projection agent。
此外，视情况而定，assets-ui agent、Claude code主agent也会参与流程。

2.各个agent之间通过带有收件目标agent的文件邮箱实现信息传递，支持一对一发送和一对多发送；当前权威实现继续使用 `.agent_runs/<round>/messages.jsonl`、`inboxes/`、`intents/` 和 `artifacts/`，不依赖 Claude Code 原生 task/subagent 通信机制。大部分agent都可以向系统里的任何agent发送消息，但player agent、character agent只能向GM/subGM agent发送消息。

3.input-analyst agent作为整个系统的入口，负责对玩家的双通道输入作初步分析，并自行决定给一个或多个agent发送消息，允许向系统里的任何agent发送消息。一般情况下，他会将角色通道的消息直接发送给GM agent，然后对用户指令通道的消息作分析处理，之后向对应的agent灵活转达。比如说涉及到系统设计之外的功能，会评估后发送给claude code主agent进行编码实现；如果是玩家主动请求UI/插图更新，会主动请求assets-ui agent更新存档级UI数据创作图片；如果玩家想要直接修改某个存档内数据，也将由input-analyst agent尝试直接读写文件完成；如果是给出的设定与前文的剧情推进发生矛盾，可以与story agent进行咨询交流后，从story agent建议的回滚点开始进行回滚操作，并要求GM agent从回滚点开始，逐一完成每一轮剧情的重新推演和输出，直到最新轮次。这时，GM agent、story agent、critic agent将在完成每一轮工作后，再开始进行下一轮的工作，保证每一轮的创作质量不受影响。

4.GM agent是世界知识的维护者、角色扮演的专业主持人、主线剧情的推进负责人，拥有游戏的全知上帝视角（可根据需要读取存档内任意文件），负责对世界观设定、剧情事件、权威客观知识、角色设定等进行读写维护更新，并担任剧情旁白，同时扮演所有非重要角色。但GM agent无权扮演玩家或任何一名重要角色，GM需要像DND跑团游戏的主持人一样，读取该actor的个人记忆文件，并以该actor主观第二人称视角向对应的player/character agent发送自然语言消息（禁止向player/character agent发送出戏的消息，也禁止发送结构化数据）。

5.GM agent可以根据玩家指令或剧情需要，将某一角色从非重要角色提升为重要角色。GM/subGM agent禁止扮演重要角色，这些必须由对应的player/character agent负责扮演。GM/subGM agent不能代替actor做任何主动行动、选择、思考、台词，但可以主动描写并告知actor感官信息，包括视觉、味觉、痛觉、触觉等内容。GM agent还可以根据玩家指令或剧情需要，更改角色的记忆、人设、背景等信息。subGM agent无权实现上述功能，但subGM agent可以向GM agent发送消息请求协助。此外，GM和subGM，以及subGM之间可以自由发送消息进行交流。

6.GM agent有义务向subGM agent询问支线剧情的推进情况，并进行指导。当GM agent已经完成了主线剧情的推进，但仍有subGM agent在推进支线剧情。GM agent应主动了解支线剧情的情况，并指导其快速完成剧情任务，等待其完成，或是直接放弃该支线。支线完成后，GM应及时将支线同步到世界知识中。

7.GM agent完成所有主线、支线剧情的推进后，将本轮的原始剧情文本（以第三人称视角呈现，忠实记录本轮的所有剧情，穿插着player/character agent的原始行动文本）发送给story agent。story agent是专业的作家，将根据写作上下文，对本轮的原始剧情文本进行加工，制作为玩家第二人称视角的小说文本，对文风进行润色的同时，保证小说文本与前文的自然衔接。并非所有原始剧情都需要在小说文本中体现，尤其是player agent认知外的支线剧情。必要的话，story agent也可以请求projection agent帮助其进行player agent视角的审核。

8.subGM agent是GM agent的助理、支线剧情的负责人，同样拥有游戏的全知上帝视角（可根据需要读取任意文件），担任支线剧情的剧情旁白，同时扮演所有非重要角色。当GM agent判断当前主线剧情发生的同时，另一地点发生了支线剧情，则新建subGM agent并指派支线剧情的相关背景、大纲、参与角色列表。subGM agent将负责担任GM推进该支线剧情，同样可以根据需要，向参与剧情的character agent发送消息。支线剧情与主线剧情一样会被持久化，且作为GM agent的后续剧情推进参考、 story agent的写作参考。GM agent只能同时存在一个实例，但subGM agent允许同时存在多个实例，以满足多个支线并行进行的情形 。同一角色（无论是重要角色还是非重要角色）一般不可参与并行发生的多个剧情。

9.projection agent负责担任GM/subGM消息的审核员。由于player/character agent作为游戏的沉浸参与者，只拥有主观视角，在其读取消息之前，projection agent将作为兜底，读取actor的设定、记忆等文件，并对GM/sub GM的消息进行审核/局部编辑/协商改写，确保角色收到的消息不破坏角色的沉浸性。局部编辑的前提是编辑范围不大，且不破坏GM/subGM的消息原意，若无法做到，应与GM/subGM协商对消息进行改写；而大部分情况下，若GM/subGM agent发出的消息已满足要求，则projection agent在审核后无需任何改动。

10.player/character agents是游戏的沉浸参与者。它们的提示词中被写入扮演的身份，从基本设定文件中读取人格，从记忆文件中读取长期/重点/短期记忆，并且只能靠自己的主观感官去感知世界。它们需要向GM agent沟通，以实现行动、感知、对话。player和character agent，以及character agents之间都不能直接互相通信，必须经由GM/subGM agent转达。更准确的说，它们只被允许向GM/subGM agent发送消息，因此，他们的提示词中也无需包含GM/subGM，无需提供文件邮箱的接收方，角色所有的输出将会直接传递给当前与角色通信的GM agent。每次行动的同时，与GM/subGM的本轮对话（包含GM/subGM发来的消息和角色发出的消息，不包含其他内容）会在actor完成回复后成对进入角色短期记忆；短期记忆不记录尚未被actor处理完成的单向GM/subGM消息，格式使用“记忆的回声：”和“我：”，并以空行分段。每轮输出结束后，系统将唤起本轮参与过剧情的player/character agents，让角色进行记忆文件的自我整理，将短期记忆转化为长期/重点记忆。短期记忆：本轮剧情过程中的记忆，在上下文中完整注入，但每轮输出结束后就需要整理，然后清空。长期记忆：角色在游戏中长期积累的记忆，在上下文中完整注入，但存在1000字的最大字数限制。重点记忆：角色在游戏中积累的对关键内容的记忆，在上下文中仅注入模糊的回忆（体现为回忆的简介），类似skill的渐进式披露机制，需要角色主动回忆，以在本轮剧情输出结束前将该段记忆保持完整注入上下文。

11.player agent和character agent的唯二区别：
一是player agent的关键行动将触发本轮剧情推演提前结束；这不由player agent自身进行判定，而是由GM agent负责判定。具体来说，当player agent向GM agent发送消息后，后续GM agent读取到该条player agent自然语言回复，并认为这个行动极为关键、足以引起剧情走向大幅变化时，才会立即提前结束本轮剧情推演；GM agent不得在尚未调用player agent时触发该行为，也不得在同一条GM输出中一边调用player agent一边预先声明`player_decision`。player agent刚刚作出的行动不会正式进入存档，但是会自动送往postprocess agent，作为本轮提供给玩家的三个行动建议选项之一。
二是player agent不参与subGM agent负责的支线剧情，这同样不由player agent判定。
除上述两点之外，player/character agent其他地方都相同，采用相同的持久化机制，存储在同一目录下。事实上，玩家甚至可以在游戏中途，通过系统指令切换扮演角色，这不需要显式写入系统流程，要相信多agent架构的自由度

12.critic agent是交付到前端之前的最后一道关卡，也是负责质量把控的唯一关卡。critic agent将分析小说文本字数是否达到要求，但因player agent关键行动导致字数不达标可豁免。此外，critic agent还将作为专业评委，评估文本的上下文连贯性、文本质量、用户指令满足度等等。若发现问题，且评估认为问题较易解决，则可进行自我修复流程。critic agent可与story agent自由交流，若协商认为是写作问题，则由story基于修改建议和本轮的原始剧情文本，重新完成小说文本的写作；若协商认为是原始剧情问题，则将存档恢复到本轮开始之前，并要求GM根据建议重新开始本轮的剧情推进；若协商认为是系统设计问题，则将请求claude code主agent进入源码修复流程。

13.postprocess agent在常规创作流程中位于critic agent之后，delivery gate之前，专门负责前端中正文以外部分的数据支持，比如当前目标、剧情摘要、下一步行动建议（当player agent本轮末尾尝试做出关键行动时，该行动将作为行动建议选项之一）、状态面板、MVU变量更新等。这部分的数据要求将可能依据存档变化。例如，当assets-ui agent更改了UI时，postprocess agent需要为更新后的UI元素提供数据支持（若UI需要数据）；若扩展UI数据提供失败，应进入异步的repair重试流程，不阻塞正文交付。

14.assets-ui agent负责调用图片生成模型，以及对前端UI的存档级别热更新；这些操作由于比较耗时，一般在每轮critic agent输出完成后，再由assets-ui agent进行异步调用；调用完成后，再实时更新到前端UI。进行图片生成时，一般需要将本存档已有的图片资源（如角色卡提取的png图片，或历史已生成的图片资源）作为参考，以确保画风、角色形象、场景风格的一致性。

15.保留 Claude Code 主 agent 作为 `/rp` 入口、底层脚本编排者，但其他运行时agent不再基于Claude CLI subprocess 执行，也不依赖 Claude Code 原生 task/subagent 机制。默认 live path 由 Python runtime 通过 `llm_runner.run_llm_agent()` 直接调用 LLM provider；

16.LLM provider支持 CC switch本地反代、OpenAI-compatible 文本 API，前者优先。cc_switch 只配置 `enabled` 与 `service_url`；其 API key、模型名和上游 provider 映射由 CC Switch/Claude Code 本地配置维护，不在本项目配置页保存。cc_switch 请求使用 300 秒 HTTP 等待窗口，以容纳长上下文 RP agent 响应；未显式覆盖时请求 `max_tokens` 为 800000。OpenAI-compatible 文本 API 配置包含 `enabled`、`base_url`、`api_key`、`model`。
CC switch反代、OpenAI-compatible 文本 API 和图片生成 API 统一由前端 API 设置页、三组环境变量和本地配置文件管理，优先级为：前端设置 > 环境变量 > 本地配置文件。前端 API 设置页写入 `skills/styles/llm_settings.frontend.json`；本地兜底配置统一写入 `skills/styles/llm_settings.local.json`。三类配置的字段优先级统一为：前端设置 > 环境变量 > 本地配置文件；环境变量分组为：`AIRP_CC_SWITCH_ENABLED` / `AIRP_CC_SWITCH_SERVICE_URL`，`AIRP_OPENAI_COMPATIBLE_ENABLED` / `AIRP_OPENAI_COMPATIBLE_BASE_URL` / `AIRP_OPENAI_COMPATIBLE_API_KEY` / `AIRP_OPENAI_COMPATIBLE_MODEL`，以及 `AIRP_IMAGE_GENERATION_BASE_URL` / `AIRP_IMAGE_GENERATION_API_KEY` / `AIRP_IMAGE_GENERATION_MODEL`。

17.每轮输入被处理之前，系统需要对存档文件夹进行自动备份；备份的存档将用于玩家主动在前端回滚历史消息、编辑历史消息后点击"更新并提交"、critic agent发起的剧情重新推演、input-analyst agent发起的设定冲突回滚等等情形。

18.除非用户在指令通道进行了特别要求，现在用户在角色通道的输入默认为第一人称，在指令通道的输入默认为第三人称，而最终输出的文本默认为用户角色的第二人称；GM/subGM agent在向player/character agent发送消息时，使用对应角色的第二人称视角；除了GM/subGM agent发来的消息以外，player/character agent读取的其他上下文，应该以该角色的第一人称视角呈现，只能使用自然语言，不允许注入结构化数据，以保证沉浸感。
角色上下文中可实现工具/协议的主动调用，但需要以自然语言方式进行，目前包括重点记忆的回忆、每轮结束后记忆的整理。比如重点记忆读取的提示词：“有些记忆一时间有些模糊，只记得大概。如果觉得现在需要想起来，可以想想：‘我想回忆：xxx’，其中xxx是这段记忆的主题词，也许就能想起来了”；记忆整理的提示词仅在轮次结束后的整理阶段给出，且记忆整理的过程不进入短期记忆：“现在我需要整理一下我的记忆，以帮助我理清思路。我的长期记忆是...，上限1000字，我的重点记忆是...，可按主题关键词进行检索，我当前的短期记忆是...。我现在需要清空短期记忆，并将长期记忆更新为... 重点记忆中，xxx片段需要更新为...，xxx片段我认为可以忘记，另外需要增加xxx重点记忆”记忆整理过程同样需要注入角色的背景、人格等信息。
此外，GM/subGM agent在向player/character agent发送消息时，必须使用第二人称的自然语言，而不是结构化数据。比如，应该使用自然语言描述当前场景、事件、角色状态、角色对话等，而不是使用结构化数据表示这些信息。

19.每个agent调用的各个功能，如果较为固定，应尽量编写为MCP工具；如果较为灵活，则以skill的形式给出。

24.player/character agent采用统一的提示词模版。目前他们有两套提示词模版，即回合内互动的提示词，位于skills\agent_prompts.py，包含回想重点记忆的协议/工具；另一套为回合结束后整理记忆的提示词，与互动提示词一样拥有回想重点记忆的协议/工具的同时，还具有部分patch更新长期记忆、全面重写长期记忆、增加重点记忆、删除指定重点记忆、更新指定重点记忆的协议/工具。调用上述协议/工具后，应在完成上下文重新注入后，再次调用该actor agent，直到agent不再调用协议/工具；注意，短期记忆是每次互动后由python脚本自动写入的，角色需要基于短期记忆更新重点记忆和长期记忆，整理结束后短期记忆会再由python脚本自动清空。memory_summaries为冗余设计，已弃用；

26.用户在角色通道的输入有两种：第一人称剧情引导，第一人称角色行动。有时会是上述两者的混合，由input-analyst agent负责拆分。其中，剧情引导是对接下来GM剧情推进的建议，GM agent会尝试根据该信息开始剧情推进，但GM agent有权根据需要进行修改，或进行更深入的展开；角色行动是扮演该角色对GM的直接回应，GM agent会将其视为来自player agent的消息，同等对待进行处理；剧情引导不会进入player agent的短期记忆，但角色行动会进入player agent的短期记忆；
