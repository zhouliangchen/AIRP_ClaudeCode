启动游戏的文件夹即为本次游玩的存档文件夹。

目前游戏支持四种启动方式，导致存档文件夹的内容会出现部分差异：
1.标准角色卡：png格式的角色卡文件，JSON 数据嵌入在图片的 `tEXt` 块中（keyword 为 `chara`），base64 编码。系统会使用 `import_prepare.py` 解析，得到无内嵌的纯PNG图片（可用于后续图片生成参考），和得到的json角色卡数据（name、description、personality、first_mes、scenario、世界书等）
2.文本资料包启动：txt/json/md格式的文本资料包，系统会使用 `import_prepare.py` 解析，尝试对文本资料进行分析，并生成存档文件。
3.旧存档：检测到 `chat_log.json` 与 `memory/` 后，系统会继续使用旧存档，继续游戏。开发阶段的本项目不保证对旧版本存档的兼容支持。
4.空白启动：当前文件夹没有png/txt/md/json/旧存档时，系统会以空白模式启动；若当前文件夹还存在其他文件，claude code会询问用户是否清空当前文件夹后再启动。空白启动状态下，系统不会自动生成开场，而是由玩家负责开头。系统会在游玩过程中基于玩家引导和剧情驱动逐渐形成并更新迭代存档文件。

<characters文件夹>(根目录/characters，不是memory/characters) 
characters/<角色名>/profile.md  角色的基本设定，与memory/characters/<角色名>/profile.md同步更新，内容保持一致；区别在于此文件为角色的第一人称口吻自我介绍。此文件由GM agent维护，供player/character agent读取；player/character agent不能修改此文件；

characters/<角色名>/long_term_memories.md 角色的长期记忆，由player/character agent自行维护；记录着角色的背景、经历、关系、目标等，作为角色的常驻上下文；上限为1000字，player/character agent在回合结束整理记忆时需要注意控制字数；否则会触发hook，提示长期记忆字数过多，要求角色重新整理；

characters/<角色名>/key_memories.json 角色的重点记忆，由player/character agent自行维护；以关键词标签索引的形式记录着角色对某件事的详细记忆，作为角色的可召回上下文；结构为记忆标签（上限20字）、记忆摘要（上限100字）、记忆详情（上限600字）；其中记忆标签、记忆摘要会常驻在上下文中，记忆详情仅在角色主动回忆时才会被注入上下文，保持到下次整理记忆为止；角色最多可同时拥有16条重点记忆；

characters/<角色名>/short_term_memories.md 角色的短期记忆，由python控制面自动维护；仅当角色收到GM/subGM消息并完成回复后，才将这一组输入输出一起写入短期记忆，避免未完成的GM/subGM单向消息提前沉淀；格式使用“记忆的回声：<GM/subGM消息>”和“我：<角色回复>”，每条消息后空行分段；短期记忆无字数限制，但每轮剧情结束后，在记忆整理完成后，系统会自动清空短期记忆；

characters/player.md 存储玩家目前操作的角色名和相对文件路径，用于映射到对应的characters/<角色名>/文件夹；格式为两行纯文本：`name: <角色名>` 与 `path: characters/<角色名>`。角色名称尚不明确时，允许以player、actor1、actor2等名称占位，并创建对应的角色存档目录；在名称明确后，input-analyst agent可根据用户指令触发角色更名。`characters/_self/` 已弃用，运行时不得再创建或读取；

</characters文件夹>
